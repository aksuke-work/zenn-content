AdSenseをツールサイトに貼るときのアプローチは、ブログやメディアサイトとは大きく違う。ユーザーはコンテンツを読みに来るのではなく、ツールを使いに来るからだ。[AND TOOLS](https://and-tools.net/) でAdSenseを最適化してきた過程で分かった、ツールサイト特有の配置戦略をまとめる。

## ツールサイトのAdSenseが難しい理由

コンテンツサイトではユーザーが記事を読む間に広告が視界に入る。スクロールしながら読むので広告との接触時間も長くなりやすい。

ツールサイトではユーザーの行動が違う：

1. ページを開く
2. 入力フォームにデータを入力する
3. 計算/変換ボタンを押す
4. 結果を見る
5. (結果をコピーして)離脱する

この流れは数十秒で完了する。ユーザーの注目は「入力フォーム」と「計算結果」に集中している。ここを外した場所に広告を置いてもほとんど見られない。

## 広告配置の原則

### 原則1: ツール機能を最優先する

広告がツールの使い勝手を妨げてはいけない。これが鉄則だ。

**NG配置の例:**
- 入力フォームと計算ボタンの間に広告
- 計算結果の上に広告（結果を見るまでにスクロールが必要）
- ファーストビューの全幅を広告が占領している

**OK配置の例:**
- ファーストビューにはツール本体のみ
- 計算結果の下に広告
- ページ末尾（関連ツールリンクの下）に広告

### 原則2: ユーザーの「止まるポイント」を狙う

ツールサイトでユーザーが一時停止するのは「計算結果を確認している瞬間」だ。この瞬間、ユーザーは画面を見て情報を処理している。広告に気づいてもらいやすいタイミングだ。

AND TOOLSでの実験では、計算結果の直下に広告を配置した場合と、ページ上部に配置した場合でCTRが2〜3倍変わった。

### 原則3: 広告密度はAdSenseポリシーを守る

Googleのポリシーでは「コンテンツより広告が多い」状態は違反になる。ツールサイトの場合、ページの情報量（ツール本体 + 解説テキスト）に対して広告が多すぎると警告を受けることがある。

目安として1ページあたり3〜4枠を上限とする。

## 広告ユニットの選択

### レスポンシブ広告 vs 固定サイズ

基本的にはレスポンシブ広告を使う。デバイスに合わせて最適なサイズが自動選択される。

ただし「計算結果の下」のような特定スポットには固定サイズを指定することもある。なぜならレスポンシブ広告はCLSを引き起こしやすいからだ（読み込みタイミングでレイアウトが動く）。

```html
<!-- 固定サイズ 300x250 のインフィード広告 -->
<div class="ad-slot" style="min-height: 250px; text-align: center;">
  <ins
    class="adsbygoogle"
    style="display:inline-block;width:300px;height:250px"
    data-ad-client="ca-pub-XXXXXXXXXX"
    data-ad-slot="XXXXXXXXXX"
  ></ins>
  <script>
    (adsbygoogle = window.adsbygoogle || []).push({});
  </script>
</div>
```

### インコンテンツ広告の活用

解説テキストが長い場合はインコンテンツ広告も効果的だ。テキストの流れに自然に溶け込む形式で、読んでいるユーザーの目に入りやすい。

## RPM改善の具体的な施策

RPM（1,000回のページビューあたりの収益）を上げるための施策を優先度順に説明する。

### 1. ジャンルの高単価化（最重要）

AdSenseの広告単価はジャンルによって大きく異なる。ツールのカテゴリがRPMに直結する。

**高RPMジャンル（目安）:**
| ジャンル | RPM目安 |
|---------|---------|
| 金融・保険・ローン | ¥800〜2,000 |
| 税金・会計 | ¥500〜1,200 |
| 法律・不動産 | ¥600〜1,500 |
| 転職・就職 | ¥400〜900 |
| IT・プログラミング | ¥300〜700 |
| 単位換算・汎用計算 | ¥100〜300 |

AND TOOLSでは税金・フリーランス関連ツールのRPMが高く、単純な単位換算ツールは低かった。これを確認してから、高RPMジャンルのツールを優先的に追加する戦略に切り替えた。

### 2. ページ別RPMを把握する

Google Analytics 4とAdSenseを連携すると、ページ単位の収益が確認できる。

連携方法：
1. AdSenseの管理画面 → サイト → 連携 → Google Analytics
2. GA4のプロパティを選択して連携
3. 数日後にGA4のレポートでページ別のAdSense収益が表示される

このデータを使ってRPMが低いページの改善を優先する。

### 3. 広告の読み込みを遅延させない

ツールサイトではページロードが速いため、広告の読み込みも早い段階で完了する。Lazy loadingを過剰に設定してスクロールしないと広告が表示されない状態にすると、表示回数がカウントされずRPMの計算が不正確になる。

ファーストビューより下の広告には適切なLazy loadingを使う：

```javascript
// Intersection Observerで広告をLazy load
const adSlots = document.querySelectorAll('.ad-slot-lazy');

const observer = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      const slot = entry.target;
      const ins = slot.querySelector('ins');
      if (ins && !ins.dataset.loaded) {
        (adsbygoogle = window.adsbygoogle || []).push({});
        ins.dataset.loaded = 'true';
      }
      observer.unobserve(slot);
    }
  });
}, { rootMargin: '200px' }); // 200px手前から読み込み開始

adSlots.forEach((slot) => observer.observe(slot));
```

### 4. 自動広告の部分的な活用

自動広告はOFFにしているが、アンカー広告（画面下部に固定表示される広告）だけはONにしている。ツールサイトではユーザーが下方向にスクロールすることが多く、アンカー広告の表示時間が長くなりやすい。

ただしアンカー広告はUXを損なうケースがある。スマートフォンで計算ツールを使っているときに画面下に広告が固定されると、操作の邪魔になることがある。この点はABテストで確認する。

## ABテストの方法

Google Optimize（サービス終了）の代わりに、シンプルなJavaScriptでABテストを実装する。

```javascript
// シンプルなABテスト実装
const variant = Math.random() < 0.5 ? 'A' : 'B';

if (variant === 'A') {
  // バリアントA: 計算結果の下に広告
  document.getElementById('ad-after-result').style.display = 'block';
} else {
  // バリアントB: ページ中央の解説テキスト内に広告
  document.getElementById('ad-in-content').style.display = 'block';
}

// GA4にカスタムイベントを送信
gtag('event', 'ab_test', {
  test_name: 'ad_placement',
  variant: variant
});
```

GA4でバリアント別のAdSense収益を比較して、RPMが高い方を採用する。

## ポリシー遵守の確認点

AND TOOLSの運営経験から、ツールサイトで注意すべきAdSenseポリシーのポイントをまとめる。

**誤クリック誘導の禁止:**
- 「広告」であることを隠すUI（ボタンに見せかける等）はNG
- 計算ボタンのすぐ隣に広告ボタンを置かない

**コンテンツポリシー準拠:**
- ツールの計算結果表示エリア内に広告を混在させない
- 計算式や結果に広告テキストが被らないようにする

**最小コンテンツ量:**
- ツールだけで計算式の説明テキストがない場合、「コンテンツが不十分」と判断されることがある
- 各ツールに200〜400文字以上の解説テキストを追加する

## まとめ

ツールサイトのAdSense最適化ポイント：

1. **配置**: 計算結果の直下が最重要スポット
2. **ジャンル**: 高RPMの金融・税金系ツールを増やす
3. **計測**: ページ別RPMを定期確認して改善対象を特定
4. **ポリシー**: 誤クリック誘導・コンテンツ不足に注意
5. **UX優先**: 広告がツールの邪魔をしない配置を徹底する

RPMは施策をしてすぐ変わるものではない。[AND TOOLS](https://and-tools.net/) でも安定したRPMになるまで3〜6ヶ月かかった。継続的に計測と改善を繰り返すことが大切だ。
